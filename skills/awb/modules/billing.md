# Billing Module

AWB 计费核心在项目组。团队积分够，不代表当前项目组能创作。

## 1. 先看什么

| 需求 | 命令 | 看什么 |
|------|------|--------|
| 创作前余额 | `points` | 团队余额、项目组余额、项目组上限 |
| 当前项目组 | `project-group-current` | `projectPointMax` 与 `projectPointBalance` |
| 消耗汇总 | `usage-summary` | 图片/视频任务、积分、人民币折算 |
| 流水排查 | `point-records` | 最近消耗 / 获得记录 |
| 充值兑换 | `point-packages` / `point-purchase` / `redeem` | 积分进团队池，不会自动分到项目组 |

默认 `1 积分 = 0.1 元`。`usage-summary` 可用参数覆盖，但常规场景不必重复传。

## 2. 经验

- 创作失败先查项目组余额和项目组上限；上限过低也会挡创作。
- 积分包和兑换码进团队池，之后通常还要调整项目组额度。
- `projectPointMax=0` 表示项目组不设独立上限，跟随团队池。
- 做 e2b / 沙箱成本核算时，优先用启动余额减当前余额；任务积分合计只能作为辅助，因为可能有未归因消耗。
- 发票申请依赖浏览器里的飞书登录态，字段多，先 dry-run。
- 充值二维码可以异步查支付状态；不要把支付等待和其他长任务混在一个阻塞流程里。
